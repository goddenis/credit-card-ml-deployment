# План A/B-тестирования моделей кредитного скоринга

## 1. Цель эксперимента

Цель — понять, какая из двух моделей лучше работает на задаче определения дефолта.

Сравниваются:
- v1 — базовая модель (LogisticRegression)
- v2 — та же модель, но с class_weight='balanced'
---

## 2. Гипотеза

Ожидается, что модель v2 будет лучше находить дефолты (выше recall и F1), но при этом может немного хуже работать по precision.

---

## 3. Разделение трафика

Трафик распределяется случайным образом:

- 50% запросов → модель v1
- 50% запросов → модель v2

Реализация:
- при отсутствии параметра `model_version` выбор модели происходит случайно на стороне API
- при передаче параметра (`v1` или `v2`) модель выбирается явно

---

## 4. Длительность теста

Тест проводится:
- либо фиксированное время (например, 2–4 недели),
- либо до достижения необходимого объема выборки.

Минимальное требование:
- достаточное количество наблюдений для статистически значимого сравнения.

---

## 5. Метрики

Основная метрика:
- F1-score — потому что учитывает и precision, и recall

Дополнительно смотрим:
- Precision — чтобы понимать, сколько лишних отказов даём
- Recall — чтобы не пропускать дефолты

С точки зрения бизнеса:
- важно не пропускать дефолтных клиентов (это прямые потери)
- но и не отказывать слишком многим хорошим клиентам

---

## 6. Сбор данных

Для каждого запроса необходимо логировать:

- входные признаки
- выбранную модель (v1 или v2)
- предсказание
- вероятность
- фактический исход (после наблюдения)

Данные используются для:
- расчета метрик
- последующего анализа

---

## 7. Анализ результатов

Для сравнения моделей можно использовать:
- t-test для сравнения средних значений
- доверительные интервалы

Это нужно, чтобы убедиться, что разница между моделями не случайная.

---

## 8. Критерий успешности

Модель v2 считается лучше, если:
- F1 выше, чем у v1
- precision не падает слишком сильно
- с точки зрения бизнеса нет ухудшения

---

## 9. Связь с архитектурой

Для поддержки A/B-теста сервис должен:

- поддерживать несколько версий модели
- логировать, какая версия обработала запрос
- обеспечивать воспроизводимость результатов

Текущая реализация:
- выбор модели через параметр `model_version`
- автоматическое распределение 50/50

---

## 10. Ограничения

- отсутствие реальных бизнес-данных для полной оценки
- упрощенная модель без глубокого тюнинга
- A/B-тест реализован на уровне API (без внешнего трекинга)

---

## 11. Возможные улучшения

- интеграция с системой логирования (ELK, Grafana)
- хранение результатов A/B-теста в базе данных
- автоматический анализ результатов
- использование специализированных платформ (например, Optimizely)
