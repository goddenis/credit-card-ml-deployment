# Возможные улучшения архитектуры

## 1. Переход от монолита к микросервисной архитектуре

Сейчас сервис реализован как одно Flask-приложение (монолит).
Внутри него находится всё: API, загрузка моделей и логика предсказаний.

Для учебного проекта это нормально:
- одна задача (предсказание дефолта)
- проще писать и тестировать
- не нужно поднимать сложную инфраструктуру

Если систему развивать дальше, этот сервис можно выделить отдельно.

По сути он уже похож на отдельный сервис инференса:
- принимает запросы
- грузит модель
- делает предсказание
- возвращает результат

---

## 2. Выделение сервиса логирования и мониторинга

Сейчас логирование почти отсутствует (или только в консоль).

В реальной системе нужно:
- логировать запросы
- записывать какая модель использовалась
- сохранять результаты

Это поможет:
- понимать как работает модель
- анализировать A/B-тест
- находить ошибки


## 3. Использование брокера сообщений для асинхронных задач

Если нагрузка вырастет, можно использовать очередь (RabbitMQ или Kafka).

Тогда:
- API кладёт задачу в очередь
- несколько воркеров её обрабатывают

Плюс:
- можно запускать несколько инстансов сервиса
- нагрузка распределяется
- система становится устойчивее
 
## 4. Хранение результатов предсказаний и A/B-тестирования

В текущей реализации результаты предсказаний не сохраняются, а возвращаются только в ответе API. Такой подход не позволяет анализировать качество модели на реальных данных и проводить полноценное A/B-тестирование.

При развитии архитектуры необходимо добавить слой хранения результатов.

Что должно сохраняться:
- входные признаки (с учетом требований безопасности и анонимизации);
- версия модели (v1 или v2);
- предсказание;
- вероятность;
- timestamp запроса;
- фактический исход (после появления данных о дефолте).

Возможные варианты хранения:
- реляционная база данных (PostgreSQL);
- хранилище событий (Kafka + downstream storage);
- data warehouse для аналитики.

Это позволит:
- рассчитывать метрики на реальных данных;
- проводить анализ A/B-теста;
- отслеживать деградацию модели (data drift / concept drift);
- формировать датасеты для переобучения.

Связь с архитектурой:
- API или worker отправляет результаты в хранилище;
- аналитические процессы рассчитывают метрики;
- результаты используются для принятия решений о замене модели.

Без этого компонента невозможно корректно завершить цикл внедрения модели в production.

## 5. Версионирование моделей и управление жизненным циклом

В текущей реализации модели сохраняются в виде файлов (`model_v1.pkl`, `model_v2.pkl`) и загружаются напрямую сервисом. Такой подход подходит для прототипа, но не обеспечивает управление версиями моделей и воспроизводимость экспериментов.

При развитии архитектуры необходимо внедрить инструменты управления жизненным циклом моделей.

### Версионирование моделей

Каждая версия модели должна иметь:
- уникальный идентификатор;
- метаданные (дата обучения, параметры, используемые признаки);
- метрики качества.

Это позволит:
- отслеживать изменения моделей;
- сравнивать версии;
- откатываться к предыдущим версиям при необходимости.

---

### Использование MLflow

MLflow может использоваться для:
- отслеживания экспериментов;
- хранения метрик (F1, Precision, Recall);
- хранения артефактов моделей;
- регистрации моделей (Model Registry).

Возможный процесс:
1. обучение модели;
2. логирование параметров и метрик в MLflow;
3. регистрация модели;
4. деплой выбранной версии в сервис инференса.

---

### Использование DVC

DVC может использоваться для:
- версионирования датасетов;
- отслеживания изменений данных;
- воспроизводимости обучения.

Это особенно важно, так как качество модели напрямую зависит от данных.

---

### Связь с архитектурой

- Training Service обучает модель и регистрирует её;
- Model Registry хранит версии моделей;
- Inference Service загружает актуальную версию модели;
- при необходимости можно переключить модель без изменения кода.

---

Внедрение этих инструментов позволяет перейти от прототипа к полноценному MLOps-пайплайну.

## 6. Масштабирование и оркестрация

В текущей реализации сервис запускается в одном контейнере через Docker или Docker Compose. Этого достаточно для учебного проекта и локальной демонстрации, но для production-среды такой подход ограничен.

При росте нагрузки сервис инференса можно масштабировать горизонтально: запускать несколько одинаковых инстансов приложения и распределять между ними входящие запросы.

### Load Balancer

Перед несколькими инстансами сервиса может быть размещен балансировщик нагрузки.

Он будет:
- принимать входящие HTTP-запросы;
- распределять их между инстансами сервиса;
- исключать неработающие инстансы из обработки;
- повышать отказоустойчивость системы.

Пример:

```text
Client
  |
Load Balancer
  |
  |--- Inference Service instance 1
  |--- Inference Service instance 2
  |--- Inference Service instance 3
```

### Kubernetes

Для оркестрации контейнеров можно использовать Kubernetes.

Kubernetes позволит:
- запускать несколько реплик сервиса;
- автоматически перезапускать упавшие контейнеры;
- масштабировать сервис по нагрузке;
- управлять конфигурацией через ConfigMap и Secret;
- выполнять rolling update при выкладке новой версии сервиса.

### Связь с текущим проектом

Текущий Docker-образ уже является основой для такого масштабирования. В дальнейшем его можно использовать как артефакт для развертывания в Kubernetes.

Возможное развитие:
- создать Kubernetes Deployment для inference-сервиса;
- задать количество реплик;
- добавить Kubernetes Service для доступа к приложению;
- подключить Ingress или внешний Load Balancer;
- настроить health-check через endpoint `/health`.

Такой подход позволит перейти от локального Docker-запуска к production-like развертыванию.
