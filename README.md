# Credit Card Default Prediction Service

---
## Описание проекта

В этом проекте реализован сервис машинного обучения для прогнозирования дефолта по кредитным картам.

В проекте показан полный цикл:
- обучение модели
- сохранение модели
- API для предсказаний
- Docker
- A/B-тестирование

Датасет: Default of Credit Card Clients Dataset (UCI).
Целевая переменная: default.payment.next.month.

## Локальный запуск

### 1. Клонирование репозитория

```bash
git clone https://github.com/goddenis/credit-card-ml-deployment.git
cd credit-card-ml-deployment
```

---

### 2. Создание виртуального окружения

```bash
python -m venv .venv
```

Активация:

Linux / macOS:
```bash
source .venv/bin/activate
```

Windows:
```bash
.venv\Scripts\activate
```

---

### 3. Установка зависимостей

```bash
pip install -r requirements.txt
```

---

### 4. Проверка наличия моделей

Убедитесь, что в папке `models/` находятся файлы:

```
model_v1.pkl
model_v2.pkl
feature_names.pkl
```

Если моделей нет — их необходимо предварительно обучить в ноутбуке из папки `notebooks/`.

---

### 5. Запуск API

```bash
python -m app.api
```

Сервис будет доступен по адресу:

```
http://localhost:5000
```

---

### 6. Проверка работы

```bash
curl http://localhost:5000/health
```

## A/B-тестирование

В сервисе реализована поддержка A/B-тестирования моделей.

Логика:
- 50% запросов направляется на модель v1
- 50% запросов направляется на модель v2
- также можно явно выбрать модель через параметр model_version

Цель A/B-теста:
- сравнить качество моделей
- выбрать оптимальную стратегию скоринга

Основные метрики:
- F1-score (основная)
- Precision и Recall (дополнительные)

Подробное описание A/B-теста, приведено в отдельном файле:

## Архитектура

### Текущее состояние (as-is)

На данный момент система реализована как монолитное приложение на Flask, включающее:
- загрузку моделей;
- обработку входных данных;
- выполнение инференса;
- API слой.

Такой подход выбран по следующим причинам:
- проект учебный и ограничен по масштабу;
- реализуется одна бизнес-функция (предсказание дефолта);
- упрощается разработка, тестирование и деплой;
- отсутствует необходимость в сложной инфраструктуре.

### Интерпретация архитектуры

Несмотря на то, что приложение реализовано как монолит, его функциональность логически соответствует отдельному сервису инференса.

При переходе к микросервисной архитектуре данный сервис может быть выделен как самостоятельный микросервис, отвечающий за:
- загрузку модели;
- выполнение предсказаний;
- обработку запросов от других компонентов системы.

Таким образом, текущая реализация может рассматриваться как прототип будущего микросервиса.

### Потенциальное развитие архитектуры

При масштабировании системы возможно выделение следующих компонентов:
- сервис инференса моделей;
- сервис логирования и мониторинга;
- сервис сбора и подготовки данных;
- сервис переобучения моделей;
- брокер сообщений для асинхронного взаимодействия.

Подробное описание возможных улучшений архитектуры приведено в отдельном документе:

[Описание возможных улучшений архитектуры](./arhitecture.md)
